# workspace-audit（工作空间审计）

## 信息架构原则：三步分流工作流

### 核心理念

`输出/` 是**过渡暂存层（Staging）**，不是永久归档。所有产出物按四叉戟分流（详见SOUL.md「三步分流工作流」章节）：

```
① 最终成果 → 归档至 源文件/ 对应分类目录（永久归档）
② 经验模式 → 沉淀至 wiki/ 对应页面（知识沉淀）
③ 运行产物 → 留在 输出/（工具/日志/DB/运行中报告）
④ 其余 → 清理删除
```

三条铁律：
1. 输出不是仓库，它是工作台。稳定后要分流，不清分就是堆积
2. 源文件只收最终版，输出的工作稿/草稿/v1-v5不归入源文件
3. wiki收可复用的洞察，不是所有输出都值得进wiki，只有模式化的经验才沉淀

### 四叉戟分类决策标准

| 去向 | 判定标准 | 典型例子 |
|------|---------|---------|
| → 源文件/ | 已完成的最终交付物（报告/方案/工作成果），不再变动 | 基线周报、工作进度表、工作思路文档 |
| → wiki/ | 可复用的经验模式、方法论、分析框架 | 会议记录格式标准、转化率诊断模型 |
| → 输出/ 保留 | 运行中的工具和产物，持续产出或引用 | 脚本(.py)、数据库(.db)、运行日志、舆情报告 |
| → 删除 | 被替代的旧版、已被wiki吸收的内容、无关项目文件 | 被v1.1替代的v1.0、已被wiki解构的知识库.md、trade-debt报告 |

### 判定优先级

```
是否可删除？
  ├─ 已被 wiki 精细化解构吸收 → 删除
  ├─ 被新版本明确替代（v1.0 → v1.1）→ 删除
  ├─ 属于无关项目（非理想之地） → 删除
  ├─ 是 pip/package 产物（site-packages/ .pyd） → 删除
  └─ 否 → 继续判断
是否需归档到源文件？
  ├─ 是最终交付物且已稳定 → 移动至源文件对应目录
  └─ 否 → 继续判断
是否需沉淀到 wiki？
  ├─ 包含可复用的模式化经验 → 提炼核心内容写入wiki页面，原文归档到源文件或删除
  └─ 否 → 保留在输出/（工具/日志/DB/进行中）
```

| 输出分类 | 源文件目标目录 |
|---------|-------------|
| 脚本工具（01-脚本/） | 长期驻留输出/（不归档） |
| 舆情系统（02-舆情/） | 长期驻留输出/（不归档） |
| 运行数据（03-数据/） | 长期驻留输出/（缓存DB不归档） |

### 三步闭环（与 meeting-minutes skill 联动）

三步闭环：源文件 → 读取+分析 → 输出 → 稳定后归档到源文件 + 同步wiki + 清理输出

Wiki 在第二步末尾更新，不等到第三步。详见 meeting-minutes skill 的三步闭环工作流章节。

### 当一个文件属于不该在输出的信号

| 信号 | 判定 | 去向 |
|------|------|------|
| 内容已被 wiki 页面覆盖 | 源文件已消化，输出可删 | 删除 |
| 是最终交付物但仍在输出 | 稳定未归档 | 移至源文件 |
| 是跨项目文件 | 错位 | 删除或移入对应项目源文件 |
| 是 pip/package 产物 | 环境依赖，非工作产出 | 删除 |
| 是纯技术/配置笔记 | 非项目产出 | 移入AIGC-KB wiki或删除 |
| 是定期报告的副本 | 输出与正式报告重复 | 删除副本，保留一份 |

- 用户要求"深度学习"某个源文件目录（如"再次深度学习01设计 使用wiki更新优化"）

### PDF处理的注意事项

| 问题 | 应对 |
|------|------|
| 大文件（>50MB） | 只读前3页快速预览，确认有文本后再全量提取 |
| 纯图片PDF（渲染图） | 无法提取文本，跳过（保留文件名索引即可） |
| 扫描件PDF | pypdf提取不到文本，需OCR（tesseract+pdf2image），但大型建筑PDF通常不是扫描件 |
| PPTX转换的PDF | 文本可提取但碎片化，需合并相邻文本片段 |
| 双栏排版PDF | 文本提取顺序可能错乱，需人工判断 |

## 扩展工作流：wiki链接审计与修复

在源文件深度学习完成后，应检查wiki页面间的链接完整性：

### 审计方法
```python
import re
pages = {}
for f in wiki.rglob("*.md"):
    if f.name not in ["index.md", "log.md", "SCHEMA.md"]:
        content = f.read_text(encoding='utf-8')
        pages[f.stem] = content

# 提取所有 [[links]]
all_backlinks = {}
for stem, content in pages.items():
    links = re.findall(r'\[\[([^\]|]+)(?:\|[^\]]+)?\]\]', content)
    for target in links:
        t = target.strip()
        if t not in all_backlinks:
            all_backlinks[t] = []
        all_backlinks[t].append(stem)

# 三类问题
orphans = [s for s in pages if s not in all_backlinks]       # 无入链
deadends = [s for s in pages if not all_links.get(s)]        # 无出链
broken = [(s, t) for s, links in all_links.items() ... ]     # 断链
```

### 修复优先级
1. **断裂链接**（指向不存在的页面）→ 先修：可能造成导航死路
2. **孤入链**（无人引用的页面）→ 从相关页面补链
3. **孤出链**（无外链的页面）→ 加"参考"章节

### 修复原则
- 断裂链指向真实存在的页面（用实际stem而非文件名）
- 孤入链从最相关的页面补引用（不要所有页面都从index链）
- 孤出链统一加 `## 参考` 章节，链到最相关的2-3个页面
- 修复后二次验证：断裂链应为0，孤入链仅剩非项目页面（如karpathy-llm-wiki）

## 前置检查

开始前，先确认：
1. 目标目录路径（默认：E:\Landofdream\输出\）
2. 读取SOUL.md中的"文件保存规范"章节，建立参照系
3. 确认用户允许的清理范围（默认：删除冗余文件/移动错位文件/重命名目录）

## 标准工作流

### 第一步：搜集现状

用Python（`os.walk`或`Path.rglob`）遍历目标目录，收集：
- 完整目录树结构（层级关系）
- 每个目录的文件数量和总大小
- 每个文件的名称、大小、修改时间
- 同时读取SOUL.md中的"文件保存规范"章节，建立参照系

输出：「目录体检报告」—— 一个顶层的全景描述 + 逐类别的数据清单。

### 第二步：问题诊断

对照SOUL.md分类和实际使用情况，识别三类问题：

| 问题类型 | 定义 | 典型例子 |
|---------|------|---------|
| 错位 | 文件放错类别目录 | 日志独立成类 → 应为报告的子目录 |
| 冗余 | 废弃文件、非工作文件出现在目录中 | pip依赖包（site-packages）不应出现在工作目录 |
| 缺失/过时 | SOUL.md列出的类别实际从未使用，或实际有类别但SOUL.md未记录 | 07-截图/ 从未使用过；07-舆情/ 实际存在但未记录 |

诊断时逐项标注：问题名称 → 具体文件/目录 → 大小影响 → 建议操作。

**文件级审计检查清单**（对每个文件逐一执行）：

```
① wiki覆盖检查：该文件内容是否已被wiki页面精细化解构？
   方法：用关键词在wiki中搜索，对比目录结构
   例：全量知识体系库(17.6KB) → wiki有46页覆盖各章节 → 删除

② 版本检查：是否存在更新的替代版本？
   方法：同名文件按修改时间排序
   例：工作表-v1.0.xlsx (8.8KB) → 有v1.1 (9.1KB) → 删v1.0

③ 脚本引用检查：数据文件/DB是否仍被脚本引用？
   方法：搜data_cache.db是否被data_fetcher.py引用
   例：data_cache.db (176KB) → data_fetcher.py有引用 → 保留

④ 格式冗余检查：相同内容多格式共存？
   方法：同名不同扩展名，取.md保留，删.docx/.html
   例：W19_基线周报.docx+html+md → 保留md，删docx+html

⑤ 跨项目检查：是否属于其他项目？
   方法：文件名/内容判断项目归属
   例：trade-deport-report → 非理想之地 → 删除

⑥ 环境产物检查：是否pip/package安装产物？
   方法：检查site-packages/ .pyd/ dist-info/目录
   例：08-依赖库/(10.7MB) → 直接删除
```

### 第三步：给出方案（关键沟通步骤）

呈现格式（先用一句话定性，再展开明细）：

```
## 当前目录结构诊断

结论：结构合理但有N个典型问题需要处理。

### 问题一：[标题]
现状描述 → 建议操作 → 理由（量化） → 预期效果
```

**二选一时给出默认走法**，不追问用户。例如：
> 06-脚本/ 和 07-舆情/ 可考虑合并，或在内部建子目录区分。**建议保持分离，07-舆情更名为07-舆情/。**

**多轮清理模式**：一次很难清理到最干净的状态。通常流程是：
- 第1轮：大扫除（删明显冗余+重命名+移动错位文件）
- 第2轮：用户提出信息架构原则 → 按原则重新分类
- 第3轮：清理遗留的中间态目录（如正式报告归档后，01-报告/只剩日志 → 日志提升一级，删除01-报告/）
- 第4轮：用户再问"还需要吗" → 检查cronjob依赖 → 最后的零头删除
每次用户提出新问题时自然触发下一轮，不要等用户说"再检查一次"。

### 第四步：执行与确认

用户确认"执行"或"可以"后：
1. 按优先级顺序执行操作：删除→合并→重命名→移动
2. 每步独立执行并输出操作结果（使用`execute_code`）
3. 全部完成后，更新SOUL.md的文件保存规范章节
4. 展示最终目录结构 vs 原始结构的对比

### 第五步：cronjob路径同步

目录重命名后，必须检查并更新cronjob中硬编码的旧路径。用 `cronjob action='list'` 列出所有定时任务，逐条检查prompt_preview是否含旧的目录路径。发现即更新。

典型更新内容：`06-脚本/ → 01-脚本/`、`07-舆情/ → 02-舆情/`、`02-数据分析/ → 03-数据/`

### 第六步：验证

- 再次遍历目录确认结果
- 确认SOUL.md已同步更新
- 如果有cronjob引用旧的目录路径，一并检查是否需要更新

## 常见的目录健康问题

1. **Python依赖库不应出现在工作目录** — site-packages/、dist-info/、.pyd 文件等是 pip install 产物，应放在虚拟环境目录下。发现即删除，需要时重新pip install即可。
2. **过小的零散快照文件** — <1KB的临时快照、缓存片段，通常可安全清理。
3. **日志独立成类** — 日志是报告的附属，应归入01-报告/下，不宜独立成09-日志/。
4. **01-报告/ 的生命周期**：正式报告全部归档后，目录只剩运行日志 → 此时应将日志提升一级（输出/运行日志/），删除01-报告/。目录本身可以消亡。
5. **目录编号跳号问题** — 经过多轮清理后，编号可能变得稀疏（02→06→07），这是因为早期有01-09个类别，删除了6个后只剩下3个。当编号不连续时，应在清理末尾阶段一次性重排为连续编号（如01-脚本/02-舆情/03-数据/），并同步更新SOUL.md和cronjob路径。
6. **SOUL.md与实际脱节** — 目录变更后必须同步更新SOUL.md，否则AI助理会依据错误的分类规范操作。

## 注意事项

- 不要在terminal中执行删除/移动，使用execute_code + Python shutil 以保证可追踪
- 每步操作后打印结果，方便用户回溯
- 删除前确认目录内容，避免误删有用文件
- 更新SOUL.md后，新对话会立即生效（下次模型加载时）
- **更新SOUL.md的可靠方法**：patch工具对外部变更敏感（会报"file modified since last read"），推荐用Python open+write直接覆盖。用replace(old_section, new_section)精确定位，写完后读文件确认。  
- **cronjob依赖检查是删除前必需步骤**。用 `cronjob action='list'` 查看所有定时任务，特别检查脚本路径引用（如 `E:/Landofdream/输出/06-脚本/xxx.py`）。有引用的文件不可删。  
- **"还需要吗"类问题先查依赖再回答**，不凭感觉判断。cronjob引用的=需要，无引用的=可清可留，交给用户决策。  
- **批次操作风险**：同时删除/移动多个文件时，执行前打印操作清单，执行后验证文件计数是否符合预期。计数不匹配（如删除11个但预期10个）说明有文件被误伤。案例：2026-05-23 session中多配置档速查.md(1.7KB)在批次删除中丢失，因文件太小未在计数中察觉，后续全盘搜索才确认。预防措施：操作前 `集合A={待操作文件}` → 操作后 `集合B={实际变化文件}` → 验证 A==B。
