# 工作空间审计实战案例：2026-05-23 理想之地输出目录清理

## 背景

理想之地项目「E:\Landofdream\输出\」目录自项目启动以来自然生长出9个分类，其中部分类别的定义与实际使用脱节，并有非工作文件混入。

## 原始目录结构

```
01-报告/     (6个文件)
02-数据分析/ (2个文件)
03-竞品分析/ (5个文件, 含监控数据/子目录)
04-策划方案/ (3个文件)
05-参考素材/ (3个文件)
06-脚本/     (11个文件)
07-舆情监控/ (18个文件, 含data/ reports/子目录)
08-依赖库/   (Python依赖包, ~10.7MB, 含lxml/ requests/ urllib3/ xhs等)
09-日志/     (2个log文件)
```

## 诊断出的问题

### 问题一：08-依赖库/ 是pip安装产物
- 内容：certifi, charset_normalizer, idna, lxml, requests, urllib3, websocket-client, xhs 的源码+编译pyd+C头文件
- 结论：不应出现在项目工作目录，应在虚拟环境site-packages中
- 操作：直接删除（约10.7MB）

### 问题二：07-舆情监控/ 命名冗余
- "舆情监控"中的"监控"二字冗余
- 且SOUL.md中对应的是"07-截图/"，从未使用过
- 操作：重命名为07-舆情/，更新SOUL.md

### 问题三：09-日志/ 不应独立成类
- 日志是报告类物的附属，独立成类会导致类别膨胀
- 操作：移入01-报告/运行日志/

### 问题四：03-竞品分析/监控数据/ 有零散快照
- 2026-05-08_首次快照.md (0.6KB) 和 _merged.md (0.5KB) 太零星
- 操作：删除两个快照，保留W19周报和README

### 问题五：SOUL.md与实际脱节
- SOUL.md列出07-截图/ 但从未使用
- 实际有07-舆情监控/ 但未记录
- 08-依赖库/ 09-日志/ 有目录但SOUL.md未列
- 操作：更新文件保存规范

## 关键经验

1. **沟通模式**：先给全景诊断，再逐项列出问题+建议+理由，用户确认"执行"后再操作
2. **二选一给默认**：不追问用户选哪个，直接说"建议走哪个方案"
3. **SOUL.md同步**：目录变更后必须更新SOUL.md，AI助理下次加载即生效
4. **顺序执行**：删除→合并→重命名→移动，每一步独立执行并输出结果
5. **最终验证**：操作完成后再次遍历确认，展示before/after对比

## 本次沉淀的信息架构原则（用户确认）

本次对话中，用户主动提出并确认了项目的**信息架构三叉戟模型**：

```
源文件/ = 永久归档层（原始资料+最终交付物）
输出/   = 过渡暂存层（加工中的半成品，稳定后清分）
wiki/   = 知识沉淀层（结构化可复用的洞察）
```

分流规则：
- 最终的交付物 → 归入源文件对应子目录
- 模式化的经验和做法 → ingest到wiki
- 其余 → 清理

这一原则已嵌入 workspace-audit skill 的「信息架构原则」章节，后续所有输出目录审计和清理操作以该原则为评判依据。
