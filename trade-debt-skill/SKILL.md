## 第一节：债权分类管理

### 1.1 按账龄分类

| 阶段 | 账龄 | 催收强度 | 对应动作 |
|------|------|---------|---------|
| 正常期 | 到期前30天 | 提示预警 | 发送付款提示函，电话确认还款计划 |
| 预警期 | 逾期1-30天 | 温和催收 | 催告函、上门面谈、登记催收台账 |
| 黄牌期 | 逾期30-60天 | 升级催收 | 律师函、启动法律评估、上报集团风控 |
| 红牌期 | 逾期60-90天 | 法律前置 | 准备诉讼材料、申请财产保全评估 |
| 法律期 | 逾期>90天 | 强制执行 | 提起诉讼/仲裁、申请保全、跟进执行 |

### 1.2 按债务人类型分类

- **国企类债务人**：优先协商化债（资产抵债/业务化债/股权置换），诉讼作为最后手段
- **民企类债务人**：同步推进诉讼保全与协商，保全措施先行
- **自然人担保**：连带责任追索、关注个人名下资产线索

### 1.3 核心监控指标

- 逾期金额及占比、账龄结构（30天/60天/90天+/180天+）
- 回款率（本期回款/应回款总额）
- 法律程序进度（保全/立案/开庭/判决/执行）
- 化债方案落地率（协议签订数/实际执行数）

## 第二节：催收策略图谱（按场景）

### 场景A：债务人经营正常但暂时性资金困难
- **策略**：分期付款协议+担保增信（追加连带担保/资产抵押）
- **关键动作**：签订《还款协议》+办理抵押登记/担保手续
- **风险点**：分期期间债务人经营恶化

### 场景B：债务人经营困难，偿债能力不足
- **策略**：以资抵债/业务化债/债权转股权
- **关键动作**：资产估值/尽职调查/协议谈判
- **风险点**：抵债资产估值虚高、流动性差

### 场景C：债务人恶意拖欠或失联
- **策略**：诉讼+财产保全+申请强制执行
- **关键动作**：证据收集（合同/结算单/催收记录）、财产线索摸排
- **风险点**：债务人已转移资产（需要前置保全）

### 场景D：债务人已进入破产程序
- **策略**：债权申报+参与债权人会议
- **关键动作**：按期申报债权、审查管理人出具的债权确认结果
- **风险点**：清偿率极低（需前置排查抵押担保）

## 第三节：化债方案评估框架

### 3.1 方案比选维度

1. **回款确定性**（高/中/低）
2. **回款周期**（3个月内/6个月内/1年内/1年以上）
3. **法律保障**（有担保/有抵押/有调解书/仅有协议）
4. **执行成本**（人力/费用/税务）
5. **合规风险**（是否涉及国有资产流失/审计问题）
6. **可操作性**（债务人配合度/资产处置难度）

### 3.2 常见方案评估（基于金瀚案例）

| 方案 | 回款确定性 | 周期 | 法律保障 | 推荐场景 |
|------|-----------|------|---------|---------|
| 强制执行（调解书） | 高 | 3-6月 | 强 | 债务人仍有可执行资产 |
| 供应链化债 | 中高 | 6-12月 | 中 | 债务人业务仍在运营 |
| 以资抵债 | 中 | 3-9月 | 中 | 资产优质、产权清晰 |
| 股权置换 | 低-中 | 12月+ | 弱 | 标的企业成长性好 |
| 业务化债（终端对接） | 中 | 6-12月 | 弱 | 债务人渠道资源好 |

## 第四节：法律程序清单

### 4.1 诉讼前准备
- 收集证据链：合同、结算单、验收单、催收记录、往来函件
- 调查债务人财产线索：银行账户、不动产、股权、应收账款
- 评估诉讼时效：普通诉讼时效3年
- 确定管辖法院：合同约定/被告住所地/合同履行地

### 4.2 申请财产保全（关键步骤）
- 保全时机：立案前/立案同时/诉讼中
- 保全范围：银行账户、不动产、股权、应收账款
- 担保方式：现金/保函/保险公司保函
- 保全期限：银行存款1年、动产2年、不动产3年

### 4.3 强制执行
- 申请依据：生效判决/调解书/仲裁裁决（需附带生效证明）
- 执行措施：查控存款/查封房产/扣押动产/限制高消费/列入失信名单
- 执行异议：债务人可能提出执行异议（需准备应对方案）

### 4.4 破产程序中债权申报
- 申报期限：债权申报期（通常为法院公告后30-50天）
- 申报材料：债权申报表、证据材料复印件、身份证明
- 表决权利：有财产担保债权/普通债权/劣后债权

### 5.6 化债方案归属核查规则（重要）

**⚠️ 2026.05.14 实测教训：** 龙陶资产抵债（武汉华生城市广场/成都春熙路商业资产）属于**苏合集团债权**的解决方案，不是上海理业的。本技能曾将该方案错误归入上海理业方案库，被用户当场纠正。

**归属核查五问（生成任何化债方案建议前必须回答）：**

| 问题 | 示例（龙陶案） |
|------|-------------|
| ① 该方案对应的债权本金来自哪个债务人？ | 苏合集团（水果贸易逾期~1,395万） |
| ② 方案涉及的资产/股权由谁持有？ | 龙陶商贸（苏合集团关联代偿方） |
| ③ 该资产/股权与哪个债权的合同/担保关系对应？ | 苏合集团代偿协议 |
| ④ 方案是否被其他债务人也用作解决方案？ | 否，龙陶仅对苏合承担代偿责任 |
| ⑤ 如果记混了，后果是什么？ | 化债优先级错配、汇报材料事实错误、领导层决策依据偏差 |

**陷阱：** 多债务人、多方案并行推进时，容易把方案A（如龙陶资产）错配到债务人B（如上海理业）。**生成任何跨债务人方案建议前，必须先查 SOUL.md 化债方案选项库或对应债务人档案，确认归属正确再输出。**

## 第九节：综合研究报告编制

### 9.1 适用场景
当需要对债务人提供的股权化债/资产抵债方案进行深度研判，需同时产出行业研究+标的分析+方案评估+汇报材料四合一报告时使用。

### 9.2 标准五章结构
| 章节 | 内容 | 前置条件 |
|------|------|---------|
| 第一章：行业全景扫描 | 市场规模、驱动因素、格局、催化剂 | 需通读行业报告 |
| 第二章：标的深度解析 | 公司概览、股权/财务/竞争定位 | 需读取公司介绍+尽调报告 |
| 第三章：化债方案评估 | 五维量化评估+多方案对比 | 需完成前三章调研 |
| 第四章：投资价值研判 | 亮点、风险、估值锚定 | 有投资决策意图时含 |
| 第五章：汇报备稿 | 正式公文格式的情况汇报 | 需向集团/审计汇报时含 |

### 9.3 化债方案五维评估量化表
| 维度 | 权重 | 评分标准 |
|------|------|---------|
| 回款确定性 | 25% | 5=有强制执行依据；3=有协议+担保；1=仅有口头承诺 |
| 回款周期 | 20% | 5=3个月内；3=1年内；1=3年以上 |
| 法律保障 | 20% | 5=已生效法律文书+保全；3=经公证协议；1=框架协议 |
| 执行成本 | 15% | 5=零成本（已有判决）；3=中等投入；1=高投入 |
| 合规风险 | 20% | 5=完全合规；3=需前置审批；1=涉及国资流失风险 |

综合评分：≥4.0优先推进，3.0-3.9审慎推进，2.0-2.9备选，<2.0不推荐。

### 9.4 汇报备稿标准模板
```
一、基本情况         → 背景/金额/合同依据

#### 首选路径：Markdown→Word (.docx) —— 使用 python-docx

```python
from docx import Document
from docx.shared import Pt, Cm, RGBColor
from docx.enum.text import WD_ALIGN_PARAGRAPH
from docx.enum.table import WD_TABLE_ALIGNMENT
from docx.oxml.ns import qn, nsdecls
from docx.oxml import parse_xml

doc = Document()
style = doc.styles['Normal']; f=style.font; f.name='SimSun'; f.size=Pt(10.5)
style.element.rPr.rFonts.set(qn('w:eastAsia'), 'SimSun')
for s in doc.sections:
    s.top_margin=Cm(2.54); s.bottom_margin=Cm(2.54)
    s.left_margin=Cm(3.17); s.right_margin=Cm(3.17)

# 工具函数
def shd(cell, color):
    cell._tc.get_or_add_tcPr().append(
        parse_xml(f'<w:shd {nsdecls("w")} w:fill="{color}" w:val="clear"/>'))

def table(headers, rows):
    t=doc.add_table(rows=1+len(rows), cols=len(headers))
    t.style='Table Grid'; t.alignment=WD_TABLE_ALIGNMENT.CENTER
    for i,h_text in enumerate(headers):
        c=t.rows[0].cells[i]; c.text=''
        r=c.paragraphs[0].add_run(h_text); r.bold=True; r.font.size=Pt(9)
        r.font.name='SimHei'; r.element.rPr.rFonts.set(qn('w:eastAsia'),'SimHei')
        r.font.color.rgb=RGBColor(0xFF,0xFF,0xFF)
        c.paragraphs[0].alignment=WD_ALIGN_PARAGRAPH.CENTER; shd(c,'2a5a8c')
    for ri,row in enumerate(rows):
        for ci,val in enumerate(row):
            c=t.rows[ri+1].cells[ci]; c.text=''
            r=c.paragraphs[0].add_run(str(val)); r.font.size=Pt(9)
            r.font.name='SimSun'; r.element.rPr.rFonts.set(qn('w:eastAsia'),'SimSun')
            if ri%2: shd(c,'f5f7fa')
    return t

def heading(doc, text, level):
    h=doc.add_heading(text,level)
    for r in h.runs:
        r.font.name='SimHei'; r.element.rPr.rFonts.set(qn('w:eastAsia'),'SimHei')
        sz=[16,13,11][level-1]; r.font.size=Pt(sz)
        r.font.color.rgb=RGBColor(0x1a,0x3a,0x5c)
    return h

def para(doc, text, bold=False, sz=10.5, color=None, align=None):
    p=doc.add_paragraph(); r=p.add_run(text)
    r.font.name='SimSun'; r.element.rPr.rFonts.set(qn('w:eastAsia'),'SimSun')
    r.font.size=Pt(sz); r.bold=bold
    if color: r.font.color.rgb=color
    if align: p.alignment=align
    return p

doc.save('/mnt/e/.../报告名称_YYYYMMDD.docx')
```

**关键样式规范：** 一级标题SimHei 16pt深蓝、二级13pt、三级11pt；正文SimSun 10.5pt；表格表头深蓝底白字9pt、数据行交替灰底；页边距上下2.54cm左右3.17cm（A4标准）。

### 10.4 XLSX 备选解析方案（无 openpyxl 时）

当系统因 PEP 668 或其他限制无法安装 openpyxl/pandas 时，利用 xlsx 本质为 ZIP 包（内含 XML 文件）的特性，用 Python 内置库直接解析：

**核心原理：** xlsx = ZIP包，内含 `xl/sharedStrings.xml`（字符串池）+ `xl/worksheets/sheetN.xml`（每张表数据）

```python
import zipfile, xml.etree.ElementTree as ET

with zipfile.ZipFile('文件.xlsx', 'r') as z:
    # 1. 读取共享字符串池
    ss_xml = z.read('xl/sharedStrings.xml')
    ss_tree = ET.fromstring(ss_xml)
    ns = {'s': 'http://schemas.openxmlformats.org/spreadsheetml/2006/main'}
    shared_strings = []
    for si in ss_tree.findall('.//s:si', ns):
        texts = [t.text for t in si.findall('.//s:t', ns) if t.text]
        shared_strings.append(''.join(texts))

    # 2. 读取指定sheet（如sheet1 = 资产负债表）
    sheet_xml = z.read('xl/worksheets/sheet1.xml')
    sheet_tree = ET.fromstring(sheet_xml)
    rows = sheet_tree.findall('.//{http://schemas.openxmlformats.org/spreadsheetml/2006/main}row')

    def get_val(cell):
        cell_type = cell.get('t', '')
        ve = cell.find('.//{http://schemas.openxmlformats.org/spreadsheetml/2006/main}v')
        if ve is not None and ve.text:
            if cell_type == 's':
                return shared_strings[int(ve.text)]
            return ve.text
        return ''

    for row in rows[:30]:  # 前30行
        cells = row.findall('{http://schemas.openxmlformats.org/spreadsheetml/2006/main}c')
        vals = [get_val(c) for c in cells]
        print(' | '.join(vals[:8]))
```

**已知局限：** 无法处理合并单元格、复杂公式计算值（只能读存储值）、数据透视表、图表。优先用 openpyxl / pandas，本方案仅在环境受限时作为备选。

### 11.4 证据矩阵框架

> **来源：** 整合自《财务尽职调查：全流程方法与实务案例》（周涛）和《投资尽职调查》（戚威）的证据层级与交叉验证体系。
> **参考文件：** `references/evidence-matrix.md` — 完整格式模板与交叉验证路径

在投资尽调报告中，将每个核心判断从六维结构化呈现，替代模糊的"我们认为"式结论：

**证据层级（质量从高到低）：** L1 官方原始文件（审计报告/银行流水/纳税申报/合同/许可证）→ L2 第三方确认（客户/供应商访谈/工商查册）→ L3 内部系统导出（ERP/CRM/订单台账/生产记录）→ L4 管理层报表→ L5 管理层访谈/BP→ L6 媒体/传闻

**证据矩阵表格标准格式：**

| 核心主张 | 证据 | 来源层级 | 验证状态 | 证据缺口 | 对交易的影响 |
|---------|------|---------|---------|---------|------------|
| 公司声称2025年营收X亿元 | 管理层收入台账 | L4 | 待验证——需与增值税申报表核对 | 未取得税务端数据 | 若无法验证需下调估值或设earnout |
| 在手订单Y亿元 | 客户签字PO扫描件5份 | L2 | 已确认——电话访谈3家客户均确认PO有效 | 未覆盖全部客户 | 支持收入预测，关注客户集中度 |
| 核心专利自主持有 | 专利证书扫描件3份 | L1 | 待法律确认——需核查发明人归属 | 未取得专利局查册报告 | 需作为交割先决条件 |

**交叉验证路径：** 收入验证（合同→验收→发票→回款）、毛利率验证（产品结构→采购成本→同业比对）、现金流验证（EBITDA→经营现金流→营运资金→自由现金流）、市场验证（TAM→SAM→SOM→竞品份额）

**报告位置：** 投资研判备忘录第二章"核心证据与假设"，在"投资亮点"之前呈现。

**三表闭环验证（进阶）：** 利润表、现金流量表、资产负债表三表互证，是判断财务质量的终极方法。每次财务分析，必须回答以下闭环问题：

| 步骤 | 问题 | 检测方法 |
|------|------|---------|
| ① 利润→现金 | 净利润变成了什么？收到了多少现金？ | 净现比（OCF/净利润），收现比（销售回款/收入） |
| ② 现金→资产 | 收到的现金/融到的资金变成了什么资产？ | 现金去向：流动资产（应收/存货）还是长期资产（固产/在建） |
| ③ 资产→利润 | 这些资产是否产生了对应的利润？ | 资产周转率、资产减值、固定资产/存货 vs 收入的关系 |
| ④ 融资→偿债 | 借来的钱能否赚回来并偿还？ | 利息保障倍数、经营活动现金流/利息支出、现金流/到期债务 |

**三表不闭的核心警示：** 利润增长但经营现金流持续为负（如宝钺案-166%毛利率+OCF为负）→利润无现金支撑；现金增长依靠借款而非经营（存贷双高）→融资驱动而非经营驱动；固定资产扩张但收入不增长→资产效率下降。

### 11.6 风险→条款映射表

> **来源：** 整合自《私募股权投资和尽职调查手册》（丁敏）第7章"交易条款"。
> **参考文件：** `references/risk-to-term-mapping.md` — 完整映射表 + 核心条款审查清单 + 宝钺案特别关注项

在条款谈判阶段，将尽调发现的风险转化为具体条款保护：

| 风险类型 | 条款应对方案 | 优先级 |
|---------|-------------|--------|
| 收入/EBITDA不确定性 | QoE调整、低倍数、earnout、托管、交割后账目对账 | 必备 |
| 营运资金缺口 | 营运资金peg、购买价调整、现金零债务机制 | 必备 |
| 税务或合规敞口 | 赔偿、托管、专业审查、交割先决条件 | 必备 |
| 客户集中度过高 | 估值折价、客户确认函作为CP、earnout、留存承诺条款 | 建议 |
| 创始人/关键人员风险 | 股权分期归属(vesting)、锁定期、竞业限制、治理权利 | 建议 |
| IP/产权瑕疵 | 交割先决条件（关闭前整改）、法律意见书、赔偿 | 必备 |
| 诉讼/环保风险 | 专业审查、托管、赔偿、排除、若无法量化则停止 | 视严重程度 |
| 管理层预测不确定性 | 调整基准案例、低估值、earnout、分期投资、里程碑式出资 | 建议 |

**宝钺案特别关注项：**
- 估值调整——以5亿元为基准价，签署交割后若发现财务差异则调价
- 实控人股权质押——增资前完成李素芬/上海芬励尔股权质押至投资方名下
- 董事会席位——至少1席，重大事项需投资方董事同意
- 优先清算权——清算/并购时优先于创始股东获本金+8%年化收益
- 交割先决条件——军工资质有效性确认、核心专利无争议、无未披露重大诉讼

### 11.11 预重整尽调适配器（新增 v1.9.2）

> **来源：** 2026-05-20 宝钺案三源综合研判（Notion #118 #119 + 华源审计 + CITIC电话会）
> **触发场景：** 尽调标的处于预重整/重整状态（如宝钺案预重整6月底完成）
> **参考文件：** `references/pre-restructuring-dd-pattern.md` — 完整核查清单+退出路径排序

#### 11.11.1 三源交叉研判模式

当三个信息源同时存在时，按以下规则交叉验证：

| 验证维度 | 优先级规则 |
|---------|-----------|
| 数字矛盾 | 审计审定数 > Notion业务记录 > 投行电话会 |
| 判断矛盾 | 拆解各方前提假设，不做简单平均 |
| 时间节点冲突 | 以最近日期的官方文件为准 |

**典型案例（宝钺）：**
- 华源审计（审定数）：良品率40%（成本核算结果）
- Notion #118（业务记录）：良品率60%（实测数据）
- **结论**：良品率处于改善趋势，40%为审计口径，60%为近期实际，报告两数并存并标注口径差异

#### 11.11.2 预重整尽调六项必查

| # | 检查项 | 红旗信号 |
|---|--------|---------|
| 1 | 预重整预计完成时间 | 无具体deadline或已延期 |
| 2 | 产业方接触数量 | <2家或仍在初期洽谈 |
| 3 | 保密协议签署状态 | 还在谈判条款 |
| 4 | 地方政府的真实态度 | 从"无力支持"转为"积极支持"即为重大正面信号 |
| 5 | 担保债务的传导路径 | 平台公司担保→代偿→形成对标的公司债权→稀释股权 |
| 6 | 设备所有权干净度 | 已做融资租赁→估值打折；未做→估值有利 |

#### 11.11.3 投资时机判断框架

```
预重整完成前（窗口期）
  → 估值尚未锁定，竞争格局未固化
  → 金瀚应跟产业方同步尽调，争取同轮次入股
  → 风险：预重整失败则投资落空

预重整完成后（竞争期）
  → 债务问题明朗，估值锁定
  → 国央企产业方竞争加剧
  → 金瀚需快速决策，可能需接受更差条款
```

#### 11.11.4 退出路径重新排序规则

预重整状态下的退出路径优先级与普通状态不同：

| 常规状态优先级 | 预重整状态优先级 | 变化原因 |
|--------------|----------------|---------|
| 上市公司并购 | 国央企产业方控股 | 整机厂可代工，实控人抗性低 |
| 独立IPO | 财务投资人接盘 | 确定性高于IPO |
| 第三方转让 | 上市公司并购 | 仍可行但谈判周期长 |
| 管理层回购 | 独立IPO | 3-5年，短期内不现实 |

#### 11.11.5 宝钺案预重整要点（示例）

| 要点 | 内容 |
|------|------|
| 预重整完成时间 | 2026年6月底（约5周） |
| 债务规模 | 建行3975万+光大4500万+供应商约4000万 ≈ 1.25亿 |
| 产业方进展 | 3-4家国央企已签保密协议，尽调同步进行 |

| 设备状态 | 未做融资租赁，所有权干净 |

### 12.2 自检→修复→复检 闭环流程

当用户要求"自检"或"评估"时，遵循以下三步闭环：

```mermaid
自检（Step 0）→ 修复（P0/P1执行）→ 复检（再次自检验证改进）
```

**三步详解：**
1. **自检**：按 11.1 五层模型全面扫描，输出自检报告（含评分和问题清单）

   **路径索引验证（每个 Layer 3 必做）：**
   ```python
   import os
   # 1. 读取 SOUL.md 中的 文件保存规范 章节，确认 `源文件\\` 前缀一致
   # 2. 读取 skill 第六节路径索引中的每条路径
   # 3. 用 os.path.exists() 检查每条路径是否可访问
   # 4. 如果催收报告等实际在 04-催收记录 而非 02-债权凭证，
   #    说明文件已被归入催收记录目录，需同步更新索引
   # 5. 记录不可达路径为 P0 问题
   ```
2. **修复**：按 P0→P1→P2 优先级逐项执行修复，每项修复后同步更新 wiki log
3. **复检**：再次运行自检流程，输出对比报告（之前评分 vs 现在评分），验证修复效果

**核心要求：** 修复后必须复检，不能默认"修好了就不管了"。复检结果需给出明确的评分变化(↑/↓/—)。

### 12.3 P0-P3 优先级治理分类

自检发现的问题按以下优先级处理：

| 优先级 | 定义 | 响应时间 | 示例 |
|--------|------|---------|------|
| **P0** | 核心目录缺失或混入无关文件，影响文件引用和归档 | 立即修复 | 01-债务人档案空仓、06-策略方案混入无关文件 |
| **P0** | SOUL.md版本与skill版本不同步（运行时SOUL路径 ≠ wiki副本路径） | 立即修复 | SOUL.md v1.3落后skill v1.9.2约3个月 |
| **P0** | 当前会话profile与知识库身份不匹配（示例：MEMORY混入"其他项目"内容） | 立即修复 | 用户身份确认 → 清理MEMORY/USER中的非相关角色信息 |
| **P1** | 数据覆盖不足，影响查询效率 | 本会话内 | 催收记录仅1份、数据分析仅1份 |
| **P2** | 元数据或索引需要微调 | 下次任务前 | SOUL.md路径描述缺项、memory容量接近上限 |
| **P3** | 自动化/前瞻性优化 | 有空时 | 设cron定时更新检查、建立自动备份机制 |

## 第十三节：跨源数据同步（Notion→Wiki）

### 12.1 适用场景

当用户要求从 Notion 读取业务记录并更新本地知识库时使用。典型触发词：\"进入我的 notion 读取...\"、\"同步 notion 数据...\"。

### 12.2 完整工作流

```
Step 1: 环境准备        → 确认 NOTION_API_KEY 可用
Step 2: 搜索目标页面     → POST /v1/search {query: "关键词"}
Step 3: 批量读取内容     → GET /v1/blocks/{page_id}/children （多页并行）
Step 4: 数据提取摘要     → 提取 rich_text/plain_text → 保存 /tmp/notion_*.json
Step 5: 保存原始数据     → 写入 09-参考素材/Notion*提取*.md
Step 6: 分析增量信息     → 对比现有 wiki 内容，标记"已有" vs "新增"
Step 7: 并行更新 wiki    → delegate_task 并行更新 entities/* 和 01-债务人档案/*
Step 8: 记录变更         → 更新 wiki/log.md
Step 9: 同步 SOUL.md     → 更新路径索引和速查表（如需要）
Step 10: 固化 memory     → 更新关键事实
```

### 12.3 API 调用模式

```bash
# 搜索
curl -s -X POST "https://api.notion.com/v1/search" \
  -H "Authorization: Bearer $NOTION_API_KEY" \
  -H "Notion-Version: 2025-09-03" \
  -H "Content-Type: application/json" \
  -d '{"query": "搜索关键词"}' -o /tmp/notion_search.json

# 读取页面内容 (最多50 blocks/次，has_more=true 时需翻页)
curl -s "https://api.notion.com/v1/blocks/{page_id}/children?page_size=50" \
  -H "Authorization: Bearer $NOTION_API_KEY" \
  -H "Notion-Version: 2025-09-03" -o /tmp/notion_blocks.json
```

**注意：** 使用 `-o` 保存文件后用 Python 解析，不要在 curl 中 pipe 到 Python（会超时）。

### 12.4 内容提取要点

```python
def extract_text_from_block(block):
    texts = []
    block_type = block.get('type', '')
    content = block.get(block_type, {})
    for field in ['rich_text', 'text']:
        if field in content:
            for rt in content[field]:
                if isinstance(rt, dict) and 'plain_text' in rt:
                    texts.append(rt['plain_text'])
    if 'children' in content:
        for child in content['children']:
            texts.extend(extract_text_from_block(child))
    return texts
```

### 12.5 增量对比策略

| 步骤 | 方法 |
|------|------|
| 对比已有 wiki 内容 | 先读 entities/对应页面，提炼已覆盖信息 |
| 标记新增信息 | 按重要性分级：重大(影响决策)/补充(修正数据)/细节(完善描述) |
| 保留已有信息 | 不要复写已有的正确内容，只追加增量 |
| 并行更新 | 使用 delegate_task 并行更新多个页面 |

### 12.7 用户声明策略变更的同步模式

区别于 Notion 数据同步的\"读取→提取→对比→写入\"流程，当用户直接口头声明策略优先级变更时，采用**直接更新**模式：

```mermaid
用户声明（如\"策略改成A→B→C\"）
  → Step 1: 识别受影响文件列表（通常5-7个）
  → Step 2: 用 delegate_task 并行更新（每次最多3个任务）
  → Step 3: 更新落下的文件（第二轮）
  → Step 4: 更新 SOUL.md 化债方案选项库
  → Step 5: 更新 wiki/log.md
  → Step 6: 更新 memory
```

**标准 7 文件清单（上海理业案策略变更时）：**

| 序号 | 文件 | 改动内容 |
|------|------|---------|
| 1 | SOUL.md | 化债方案选项库重排序号+标注优先级箭头 |
| 2 | wiki/entities/shanghai-liye.md | 方案顺序调整+当前状态表更新 |
| 3 | wiki/entities/debt-overview.md | 策略总览表更新+时间线新增决策记录 |
| 4 | wiki/concepts/collection-strategies.md | 策略矩阵+决策建议表重排 |
| 5 | wiki/comparisons/方案对比.md | 对比表+推荐路径+综合建议重排 |
| 6 | 01-债务人档案/对应债务人.md | 化债方案进展表重排 |
| 7 | 04-催收记录/对应催收日志.md | 追加决策记录 |

**关键风险点：** 文件数量多（5-7个），容易遗漏。策略是分两轮 delegate_task（每次最多3个），最后一轮补剩下的。更新完毕后，手动验证 SOUL.md + 1个 wiki 页面 + 1个源文件 三个层面各至少一个文件，确保全线覆盖。

### 12.8 已知陷阱

| 陷阱 | 表现 | 避免方法 |
|------|------|---------|
| 页面数据量大 | 单次请求仅50 blocks，大量内容未读取 | 检查 response.has_more，翻页继续 |
| 数据冗余 | 多条 Notion 页面包含相同内容 | 增量对比时按 page_id 去重 |
| 幻觉风险 | delegate_task 子任务可能编造 Notion 数据 | 校验子任务输出中的精确数值与原始 JSON 一致 |
| 读取不完整 | 只读了最新几页，遗漏历史记录 | 检查搜索结果的 total 数，按需分页读取全部 |

## 第十四节：标准化输出规范（架构可见性强制规则）

> **本节的来源：** 用户指出本智能体的输出"知识库架构没有输出"——即每次产出的结构不可见、不清晰。
> **核心原则：** 知识库架构必须**显式呈现在每次输出中**，而不是隐藏在后台逻辑里。

### 14.1 模板引用强制规则

**每次产出物第一段必须包含以下标注：**

```
模板来源：output-templates.md → T-XXX v1
风险等级：按 risk-rating-system.md X.X节 [场景分类]
```

如：
```
模板来源：output-templates.md → T-CR v1
风险等级：按 risk-rating-system.md 1.1节 合同审查分类
```

**缺少模板标注即违规。** 用户看到的第一眼就应明确知道"这份产出的框架是什么、什么结构的、怎么分类的"。

### 14.2 六种输出模板速查

| 编码 | 模板名称 | 适用场景 | 文件命名后缀 |
|------|---------|---------|-------------|
| T-CR | 合同审查意见 | 协议/合同签署前审查 | `_T-CR.md` |
| T-IDN | 投资研判备忘录 | 标的企业尽调/估值分析 | `_T-IDN.md` |
| T-COL | 催收/化债方案评估 | 催收策略建议/方案比选 | `_T-COL.md` |
| T-REP | 公文/情况汇报 | 向集团/区政府/审计正式汇报 | `_T-REP.md` |
| T-FAST | 新信息快速研判 | 电话会纪要/研究报告即时决策 | `_T-FAST.md` |

> 详见 `references/output-templates.md` 获取各模板完整结构。

### 14.3 风险等级颜色使用规范

所有涉及风险判断的产出，必须使用统一的风险标记体系：

| 标记 | 合同审查含义 | 催收评估含义 | 投资尽调含义 |
|------|-------------|-------------|-------------|
| 🔴 红线 | 不改不签 | 不推荐（回款<20%） | 低置信度 |
| 🟡 高风险 | 强烈建议修改 | 备选观察（20-50%） | 中等置信度 |
| 🟢 中风险 | 建议修改 | 审慎推进（50-80%） | 高置信度 |
| ⚪ 低风险 | 无需修改 | 优先推进（≥80%） | — |

> 详见 `references/risk-rating-system.md`。

### 14.4 风险分析"五看"框架（合同审查场景）

识别一条合同条款风险等级的标准方法：

| 步骤 | 问题 | 判定 |
|------|------|------|
| 看1 | 是否违反法律强制性规定？ | 是→🔴红线 |
| 看2 | 是否导致核心义务/权利虚化？ | 是→🔴红线 |
| 看3 | 是否显著增加本方维权成本？ | 是→🟡高风险 |
| 看4 | 是否造成本方被动接受的机制？ | 是→🟡高风险 |
| 看5 | 是否限制本方商业自由？ | 是→🟢中风险 |

### 14.5 催收方案"五维评估"量化表

| 维度 | 权重 | 5分 | 3分 | 1分 |
|------|------|-----|-----|-----|
| 回款确定性 | 25% | 有强制执行依据 | 有协议+担保 | 仅有口头承诺 |
| 回款周期 | 20% | 3个月内 | 1年内 | 3年以上 |
| 法律保障 | 20% | 已生效法律文书+保全 | 经公证协议 | 框架协议 |
| 执行成本 | 15% | 零成本 | 中等投入 | 高投入 |
| 合规风险 | 20% | 完全合规 | 需前置审批 | 涉及国资流失 |

综合评分 ≥4.0优先 / 3.0-3.9审慎 / 2.0-2.9备选 / <2.0不推荐

### 14.6 文件命名规范

正式报告（投资研判备忘录 / 综合研究报告 / 化债方案评估 / 公文汇报）默认输出 **Word (.docx)** 格式：

```
[内容简述]_[YYYYMMDD]_v[版本号].docx
```

示例：
- `湖南宝钺综合研究报告_20260514_v3.0.docx`

```
[内容简述]_[日期YYYYMMDD]_[模板编码].md
```

示例：`华源证券财务顾问协议审查意见_20260509_T-CR.md`

### 14.7 已知陷阱

| 陷阱 | 表现 | 避免方法 |
